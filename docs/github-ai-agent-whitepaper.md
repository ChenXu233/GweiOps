# GweiOps 白皮书

## 从问题到生产，全链路运维闭环的 AI 大脑


## 一、摘要

GweiOps 是一个 AI 驱动的全链路运维智能体平台。它完整接管从问题产生到生产环境验证的闭环：

**问题感知 → 诊断决策 → 方案执行 → 验证确认 → 生产分流 → 扩流/回滚 → 最终审核**

这不是七个独立的步骤，而是一条由 GweiOps 自主编排的自动化流水线。GweiOps 通过 API 对接企业现有的全部 DevOps 工具——代码托管、CI、CD、监控、通知——以事件驱动的方式串联整个链路，只在最终审核环节请求人类介入。

问题来源不限于代码仓库的 Issue。GweiOps 从安全扫描器（0day 扫描）、CVE 发布网站、监控告警系统等一切可能影响生产稳定性的信号中感知问题。解决方案不限于代码修复。GweiOps 通过插件机制扩展至基础设施运维动作——扩容、迁移、配置变更、流量调度、回滚等。

GweiOps 不替代任何现有工具。不自建 Git，不跑流水线，不调度 Pod。它是 DevOps 工具链的 AI 大脑，指挥已有工具协同完成从故障发现到生产验证的全流程。

GweiOps 平台完全开源，核心代码可在 GitHub 获取。官方提供一键式 SaaS 服务和私有化部署方案。更重要的是，GweiOps 自身的仓库和 SaaS 服务由 GweiOps 自己管理——团队是第一个也是最严格的用户。

GweiOps 奉行"人在回路"的设计哲学：AI 在紧急情况下自动执行全流程，确保服务分钟级恢复；人类在日间从容审案，确认或调整方案，并在此过程中持续训练 AI。每一次人类的退回与修正，都被 GweiOps 记录、分析、向量化沉淀为团队的集体决策记忆。

GweiOps 的名称源自《易经》：

- **既济**：事已成，秩序井然。对应 PR 合并、服务稳定运行的理想态。
- **未济**：事未成，混乱流动。对应 Issue 积压、半夜告警的现实态。

《易经》将"未济"置于最后一卦——万事万物永远在"完成"与"未完成"之间轮回。GweiOps 的使命，正是在"未济"与"既济"之间持续摆渡。

**GweiOps —— 摆渡未济，恒守既济。**


## 二、名字释义

Gwei 取自《易经》第 63 卦"既济"与第 64 卦"未济"。

| 卦象 | 含义 | 软件工程对应 |
| :--- | :--- | :--- |
| 既济 | 事已成，渡河完毕，秩序井然 | PR 合并、漏洞修复、服务稳定 |
| 未济 | 事未成，渡河未竟，混乱流动 | Issue 涌入、0day 突发、生产告警 |

"既济"是第 63 卦，"未济"是第 64 卦——宇宙的常态是"未济"，而"既济"是永远在追求却永远无法终极抵达的彼岸。

Ops 代表 Operations，DevOps、GitOps、SecOps、MLOps 皆以此结尾，代表持续不断的自动化维护。GweiOps 加上 Ops 后缀，明确宣告这是一个实打实的生产环境运维系统。

**GweiOps —— 摆渡未济，恒守既济。**


## 三、市场背景与问题陈述

### 3.1 开发团队被"运维债务"压垮

全球开发者超过 3000 万，写新代码只占工作时间的一小部分，大部分时间被维护吞噬：处理 Issue、排查故障、应对 0day、通宵值班。

Google《AI for DevOps 白皮书》（2023）指出，机器学习在日志分析、异常检测和根因分析方面展现巨大潜力。Azure 的 OpsCopilot 将 MTTR 降低了 50%。但这些工具仍停留在"辅助人类"层面——它们帮人类更快做事，但人类依然要半夜爬起来做事。

### 3.2 当前市场的三大断层

**工具链割裂。** 代码托管、CI、CD、监控各自为政，AI 工具只覆盖单点。故障发现后，人工定位、人工修复、人工部署，链路漫长。

**同步工具无法无人值守。** Claude Code、Cursor 需要人类在电脑前发起对话。凌晨三点的告警，依然要人类爬起来处理。

**平台锁定严重。** 多数工具绑定 GitHub 或单一云厂商，企业换平台等于换工具。

### 3.3 市场机遇

IDC 预测，到 2027 年，80% 的中国 500 强企业将部署代理式 AI 平台。全球 AI 编码工具市场预计从 2025 年的 76.5 亿美元增长到 2026 年的 94.6 亿美元。

GweiOps 正站在 AI Agent 与 DevOps 的交汇点上——一个即将爆发但尚未被定义的蓝海。


## 四、核心定位

### 4.1 一句话定位

GweiOps 是 DevOps 工具链的 AI 大脑，**完整接管"问题→行动→验证→生产分流→审核"的全链路闭环**，让 AI 在夜间替人类值班，人类在早晨从容审案。

问题来源不限于代码仓库的 Issue，而是涵盖安全扫描器、CVE 漏洞源、监控告警系统等一切可能影响生产稳定性的信号。解决方案不限于代码修复，通过插件机制扩展至基础设施运维动作——扩容、迁移、配置变更、回滚等。

### 4.2 GweiOps 是什么，不是什么

**GweiOps 是：**

- DevOps 工具链的 AI 编排与决策层
- 事件驱动的无人值守全链路自动化系统
- 团队决策记忆的持续学习平台
- 生产环境的容错与自愈大脑
- 人在回路的决策管理平台

**GweiOps 不是：**

- 代码托管平台（不自建 Git）
- CI/CD 引擎（不跑流水线）
- 部署调度系统（不调度 Pod）
- 监控系统（不存储指标）
- 通用 AI 编程助手（不依赖对话）

### 4.3 与竞品的根本性切割

| 维度 | Claude Code / Cursor | ArgoCD / Flux | **GweiOps** |
| :--- | :--- | :--- | :--- |
| 运行模式 | 按需唤醒 | 声明式配置 | **常驻后台，事件驱动，无人值守** |
| 作用阶段 | 编码阶段 | 部署阶段 | **维护阶段——全链路** |
| 覆盖范围 | 代码层 | 部署层 | **问题→行动→验证→分流→审核** |
| 人类角色 | 改代码 | 写配置 | **做选择题、训练 AI** |
| 价值锚点 | 提升编码速度 | 提升部署效率 | **降低恐慌指数，买回睡眠** |

### 4.4 大脑与四肢的分工

**GweiOps（大脑）** 感知事件、理解上下文、制定方案、指挥执行、汇总结果。它调用 Git API 读取 Issue、创建分支、提交 PR；调用 CI API 触发构建；调用 CD API 触发金丝雀发布、扩流、回滚；读取监控 API 接收告警、轮询指标；推送通知 API 发送报告、接收指令。

**下游工具（四肢）** 执行具体动作：Git 托管提交 PR，CI 系统执行构建，CD 系统执行部署和金丝雀，监控系统发送告警和指标，通知系统推送报告。

GweiOps 本身也是一个被 CI/CD 管理的普通应用——代码→构建→发布，生命周期由下游工具管理。但它反过来元管理这些工具，通过 API 指挥它们协同作战。

**这是双向关系：GweiOps 是 CI/CD 管理的对象，CI/CD 是 GweiOps 管理的对象。**

### 4.5 GweiOps 与 Kubernetes 的关系

GweiOps 与 Kubernetes 的关系，不是竞争，而是分层协作。Kubernetes 解决的是容器调度问题——把 Pod 放到合适的节点上运行，考虑 CPU、内存、亲和性、拓扑约束。Kubernetes 不关心 Pod 里面跑的是什么业务逻辑，它只关心资源。

GweiOps 解决的是运维决策编排问题——收到一个告警后，应该调用哪个诊断智能体来分析？诊断结果出来后，应该走代码修复还是直接扩容？修复方案有激进、保守、折中三种，选哪个？风险等级如何判定？要不要等人工确认？

Kubernetes 调度的对象是容器（Pod）。GweiOps 调度的对象是智能体任务（诊断、修复、决策）。两者完全不重叠。GweiOps 利用 Kubernetes 提供的资源底座，但处理的是 Kubernetes 完全不理解的领域：AI 模型调用、策略路由、审批流编排、团队记忆、风险判定。

| | Kubernetes | GweiOps |
| :--- | :--- | :--- |
| **调度的对象** | 容器（Pod） | 智能体任务（诊断/修复/决策） |
| **调度依据** | CPU、内存、节点亲和性 | 问题类型、代码上下文、历史偏好、风险等级 |
| **调度结果** | Pod 被分配到某节点 | 任务被路由到某智能体，产生决策输出 |
| **关系** | **底座**——GweiOps 跑在 K8s 上 | **上层**——利用 K8s，但不替代 K8s |

**GweiOps 就是运维工具链的 Kubernetes。** 正如 Kubernetes 把容器运行时、存储、网络这些底层能力缝合成一个统一的操作界面，GweiOps 把 Git、CI/CD、监控、日志、告警、K8s、Service Mesh、云厂商 API 这些工具链缝合成一个统一的运维决策界面。开发者不需要关心 Pod 跑在哪台机器上，运维人员也不需要关心在哪个工具里完成哪个步骤——一切都被缝合好了。

### 4.6 GweiOps 解决的核心问题：如何运维

GweiOps 解决的不是"运维工具"的问题，而是"如何运维"的问题。

一个几百人的运维团队之所以需要那么多人，不是因为"操作"本身复杂，而是因为知识分散在几百人脑中、决策路径冗长、新人培养周期长。GweiOps 把"怎么做"的知识从人脑转移到 AI 大脑，让运维这件事变得可控、可复制、可传承。

**一个几百人团队才敢运维的复杂系统，现在几个人 + GweiOps 就能管住。这就是 GweiOps 的核心价值。**


## 五、全链路闭环

### 5.1 问题来源

GweiOps 感知的问题来源分为四类，每类对应不同的处理策略和插件路由。

**代码仓库 Issue 类。** GitHub、GitLab、Gitee 等平台提交的 Bug 报告、功能请求、重构建议。处理策略通常为代码修复、方案生成、PR 提交。信息来源是仓库 Webhook 和 API 轮询。

**安全漏洞类。** 0day 扫描器（如 Trivy、Grype、Snyk）检测到的依赖漏洞、容器镜像漏洞。CVE 发布网站（如 NVD、GitHub Advisory Database）推送的公开漏洞信息。处理策略通常为依赖升级、补丁回溯、安全配置加固。信息来源是扫描器 Webhook、CVE 源 RSS/API。

**服务中断告警类。** Prometheus、Sentry、DataDog 等监控系统推送的生产故障告警——服务不可用、错误率飙升、延迟超限、内存泄漏、CPU 飙高。处理策略通常为代码热修复、服务重启、流量切换、实例扩容。信息来源是监控系统 Webhook。

**基础设施预警类。** 磁盘使用率超限、数据库连接池耗尽、QPS 接近阈值等尚未中断但即将引发故障的预警信号。处理策略通常为自动扩容、配额调整、配置优化。信息来源是监控系统、云厂商 API、K8s 事件。

### 5.2 解决方案：三条路径

GweiOps 诊断智能体判断问题类型后，路由到三条执行路径。

**路径一：上游更新（第一优先级）**

诊断智能体检测到问题类型为依赖漏洞或已知 CVE 时，首先检查上游是否有官方修复。

典型场景包括：漏洞扫描器发现当前依赖的第三方库存在 CVE，诊断智能体查询该库的最新版本，确认新版本已修复该漏洞，执行路径为更新依赖版本、提交 PR、触发 CI、金丝雀发布。CVE 发布网站推送新漏洞公告，影响当前使用的某个中间件，诊断智能体检查官方补丁状态，执行路径为将补丁版本写入依赖文件、提交 PR、走标准发布流程。社区已经有人提交了修复 PR 但尚未合并，诊断智能体检测到该 PR，将补丁作为临时方案引入，执行路径为生成临时补丁、提交 PR、走标准发布流程，最终审核时建议上游合并后切换回官方版本。

上游更新是第一优先级，因为上游更新是官方验证过的方案，安全性最高，引入新问题的风险最低；不需要 GweiOps 生成代码，减少 AI 幻觉带来的风险；通常只需要修改一行依赖版本号，执行效率最高；长期来看，跟随上游版本是可持续的维护策略。

**路径二：应急补丁 + 上游追踪**

上游暂无官方修复时，GweiOps 自动生成应急补丁，同时启动上游追踪。两条时间线并行执行：

- **应急修复时间线**：生成补丁（patch-package / go mod replace / 内联修复）→ 提交 PR → CI 构建 → 金丝雀发布 → 等待人类终审
- **上游追踪时间线**：后台持续轮询依赖仓库 Releases、监控 CVE 状态更新、监听 GitHub Tag 事件 → 上游发布官方修复后，GweiOps 自动生成"切回官方"PR → 进入标准发布流程 → 等待人类终审

人类在最终审核时看到的是两条时间线的独立决策：

| 时间点 | 人类决策 | 选项 |
| :--- | :--- | :--- |
| **第一次（应急补丁审核）** | 是否采纳应急补丁？ | 采纳 → 立刻恢复服务；不采纳 → 服务继续受损，等上游 |
| **第二次（上游追踪触发）** | 是否切回官方版本？ | 切换 → 移除补丁，转向官方；不切换 → 继续用补丁，等下一次版本 |

人类在第一次审核时说"我都要"——既要应急补丁先跑着不让服务停，又要确保有一天能彻底甩掉补丁回到官方主流轨道。GweiOps 替人类把两条路都走了，让人类在每条路的终点做一次独立的决策。

**路径三：非代码运维动作**

诊断智能体判断根因不在代码层时，调用对应插件执行基础设施运维动作。

典型场景包括：故障由流量过载导致，插件调用云厂商 API 或 K8s API 水平扩容 Pod 副本数；故障由数据库连接池耗尽导致，插件修改连接池配置并触发重启或滚动更新；故障由依赖服务不可用导致，插件切换流量至备可用区或触发降级开关；磁盘使用率超限，插件自动清理日志或临时文件，或触发存储扩容。

### 5.3 七步闭环

无论问题来源是什么，无论解决方案是代码还是非代码，GweiOps 的七步闭环框架保持一致。

**第一步，问题感知。** GweiOps 感知智能体从 Issue 系统、安全扫描器、CVE 源、监控告警等渠道识别问题信号。

**第二步，诊断决策。** 诊断智能体分析上下文，定位根因，判断问题类型，决策采用代码修复还是非代码插件方案。

**第三步，方案生成与执行。** 若为代码类，生成修复代码并提交 PR；若为非代码类，调用对应插件执行运维动作。

**第四步，验证。** 观测智能体持续轮询监控指标，判断问题是否解决、系统是否恢复健康。

**第五步，生产分流。** 若涉及新镜像，执行金丝雀发布；若为非代码动作，按插件策略执行灰度操作。

**第六步，扩流或回滚。** 指标正常则扩大范围；指标异常则自动回滚代码或撤销运维动作。

**第七步，最终审核。** 推送完整报告至通知系统，等待人类终审确认。所有方案思路和决策依据向量化入库。

### 5.4 完整工作流示例一：凌晨 3 点内存泄漏告警

**① 问题感知（03:17）。** Prometheus 检测到内存泄漏告警，Webhook 推送至 GweiOps。

**② 诊断决策（03:18–03:20）。** GweiOps 诊断智能体分析日志和代码上下文，定位根因为某函数未释放资源。决策采用代码修复方案。

**③ 方案生成与执行（03:21–03:26）。** 修复智能体生成三套方案（激进/保守/折中），通过 API 切分支、提交 PR，触发 CI 构建。

**④ 验证（03:27–03:32）。** 新镜像构建完成，观测智能体准备接管验证。

**⑤ 生产分流（03:33）。** GweiOps 指挥 CD 执行金丝雀发布，切 1% 流量至新版本。

**⑥ 扩流或回滚（03:34–03:45）。** 1% 稳定后扩至 10%，稳定后扩至 50%，最终扩至 100%。全程自动，任何阶段异常则回滚。

**⑦ 最终审核（08:00）。** GweiOps 推送飞书消息："昨夜 03:17 内存泄漏已自动修复，金丝雀发布成功，请终审确认。"

**关键指标：从故障发生到服务恢复，全程不超过 10 分钟，人类全程无需惊醒。**

### 5.5 完整工作流示例二：凌晨 3 点流量过载告警

**① 问题感知（03:17）。** Prometheus 检测到服务 A 的 QPS 超限，错误率飙升至 15%，Webhook 推送至 GweiOps。

**② 诊断决策（03:18–03:20）。** GweiOps 诊断智能体分析指标和日志，判断根因为流量突增导致 Pod 资源饱和，非代码逻辑问题。决策调用扩容插件。

**③ 方案执行（03:21–03:24）。** 扩容插件调用 K8s API，将服务 A 的副本数从 3 扩容至 10。滚动更新过程中新 Pod 逐步加入服务网格。

**④ 验证（03:25–03:30）。** 观测智能体持续轮询 Prometheus，错误率从 15% 降至 0.2%，延迟恢复正常。

**⑤ 生产分流与稳定。** 扩容动作已直接生效，新 Pod 承载流量，系统恢复稳定。

**⑥ 最终审核（08:00）。** GweiOps 推送飞书消息："昨夜 03:17 服务 A 流量过载，已自动扩容至 10 副本，错误率已恢复。请确认扩容策略是否需要调整。"

### 5.6 完整工作流示例三：依赖漏洞 + 上游追踪

**① 问题感知（03:17）。** 漏洞扫描器检测到依赖 X 存在 CVE-YYYY，Webhook 推送至 GweiOps。

**② 诊断决策（03:18–03:20）。** GweiOps 诊断智能体查询依赖 X 的上游状态，确认上游暂无官方修复版本。决策启用应急补丁 + 上游追踪双轨策略。

**③ 方案执行（03:21–03:26）。** GweiOps 自动生成应急补丁（patch-package），提交 PR，触发 CI 构建。同时启动上游追踪任务，开始轮询依赖 X 的 GitHub Releases。

**④–⑥（03:27–04:00）。** 新版本完成金丝雀发布，服务恢复。上游追踪任务持续运行。

**⑦ 第一次最终审核（08:00）。** GweiOps 推送报告："依赖 X 存在 CVE-YYYY，上游暂无修复。已生成应急补丁并完成金丝雀发布。请确认是否采纳应急补丁。" 人类决策：采纳。

**⑧ 上游追踪触发（两个月后的某天，04:30）。** 上游发布 v2.3.1 修复版本。GweiOps 自动检测到，生成"切回官方"PR，完成 CI 构建和金丝雀发布。

**⑨ 第二次最终审核（08:00）。** GweiOps 推送报告："上游已发布官方修复版本 v2.3.1，应急补丁已自动移除并切换至官方版本。请确认是否切回官方。" 人类决策：确认切换。

### 5.7 状态机设计：可回滚的工作流

GweiOps 的核心创新在于工作流状态机支持任意节点回退。

标准工作流包含七个状态，对应上述七个环节。到达最终审核节点后，人类有三种选择：

- **确认有效**：闭环完成，进入"既济"状态。修复方案和学习标签入库。
- **方案需调整**：回退至"诊断决策"或"方案生成"节点，携带人类新指示重新执行。
- **方案有严重问题**：触发代码回滚或运维动作撤销，回到故障前状态，等待人工介入。

**关键区别**：代码回滚是运维动作，决策回滚是管理动作。决策回滚不是撤销，而是一次在线训练——人类在教 GweiOps 如何更像团队的一员。

### 5.8 持续学习的记忆系统

每一次决策回滚都被 GweiOps 完整记录并向量化存储。

下次遇到同类 Issue 时，GweiOps 自动注入历史记忆，参考团队过往的决策偏好调整新方案。

这就是 GweiOps 的飞轮效应：AI 生成方案→人类退回+指示→记录偏好→下次自动注入→方案更贴近团队风格→退回率降低→人类更信任→更多决策权下放。

**别人可以复制代码，但复制不了跑了两年的团队专属决策记忆。**


## 六、人在回路

### 6.1 分层决策模式

GweiOps 奉行"紧急授权 + 日间审判"的分层决策：

- **夜间紧急模式**：AI 全自动执行七步闭环，人类安心睡觉
- **日间审案模式**：人类在 Dashboard 上审查报告、做选择题、训练 AI
- **核心原则**：AI 可以犯错，但 GweiOps 确保人类能轻松纠正，且纠正过程成为系统经验

### 6.2 人类方案的接入

GweiOps 不要求人类改变任何工作习惯。

当人类工程师希望提交自己的修复方案时，在 Git 工作流中正常提交 PR，在描述中标注关联 Issue 编号。GweiOps 自动识别并标记为"人类方案"，暂停 AI 方案，将人类方案纳入金丝雀验证流程。

同一 Issue 下，AI 多套方案与人类方案并行验证，Dashboard 上展示所有方案的对比数据——通过率、性能指标、错误日志摘要。技术负责人选择最优方案合并，所有方案思路均向量化入库。

**无论 AI 方案胜出还是人类方案胜出，经验都沉淀为团队记忆。**

### 6.3 Dashboard：态势感知

GweiOps 的前端面向技术负责人和老板的决策与监管需求。

- **一屏概览**：所有 Issue 状态、Agent 运行情况、金丝雀进度
- **数据驱动**：方案对比展示验证数据，而非代码
- **零操作焦虑**：操作在 Git/CI/CD 里完成，Dashboard 只负责呈现和确认
- **三色态势**：红/黄/绿，十秒判断整体健康度

核心面板包含四个区块：

- **整体态势**：系统状态、待审数量、昨夜修复数量、记忆库条目数
- **待审案件**：所有等待审批的 Issue，每个 Issue 展示所有可用方案及其金丝雀验证结果
- **昨夜回顾**：按时间线展示夜间自动修复的所有事件
- **系统健康**：金丝雀运行数量、成功率、平均修复时间（MTTR）


## 七、平台架构

### 7.1 大脑内置能力

GweiOps 大脑随核心一起发布，自带以下内置能力，开箱即用，不需要装任何插件。

**问题感知（内置）** ：GitHub/GitLab/Gitee Webhook 接收器、Prometheus AlertManager 集成、基础 CVE 源订阅。

**诊断决策（内置）** ：依赖漏洞诊断（检测 CVE → 查询上游修复状态 → 判断是否有官方修复版本）、代码故障诊断、基础设施诊断、决策路由（上游更新 > 自主修复 > 非代码运维）。

**上游追踪（内置）** ：依赖版本轮询、CVE 状态追踪、自动生成切换 PR。

**执行能力（内置）** ：基础 Git 操作（通过 API 创建分支、提交 PR、发表评论）、基础 K8s 操作（读取 Pod 状态、触发 HPA 扩容）、基础通知（飞书/钉钉/Slack 报告推送）。

**金丝雀发布编排（内置）** ：标准金丝雀流程（1% → 10% → 50% → 100%）、自动回滚触发。

**Dashboard（内置）** ：态势感知面板、方案对比视图。

**记忆系统（内置）** ：决策回滚记录、向量化存储、自动注入历史偏好。

### 7.2 大脑与执行工具的协作

GweiOps 大脑本身不实现任何基础设施层面的执行能力，而是通过 API 对接执行平台来完成运维闭环。大脑解决的是"如何运维"的决策问题，执行平台解决的是"运维动作"的具体执行。

| 能力 | 由谁执行 | GweiOps 的角色 |
| :--- | :--- | :--- |
| Pod 扩缩容 | Kubernetes HPA / K8s API | 观测到指标异常 → 调用 K8s API 调整副本数 |
| 连接排空 | Kubernetes preStop hook + Service | 发布前 → 确保 K8s 执行了排空 |
| 会话粘滞 | Istio / Service Mesh | 金丝雀发布时 → 配置路由规则 |
| 数据库迁移 | 客户的迁移脚本 | 发布前 → 执行迁移 → 等待完成 → 再切流量 |
| 缓存预热 | 客户业务代码 | 新 Pod 启动 → 触发预热 → 确认就绪 → 接入流量 |
| 依赖服务检查 | 客户健康检查接口 | 金丝雀验证时 → 轮询检查 → 确认健康 → 继续扩流 |
| 流量切换 | Ingress / Service Mesh / LB | 金丝雀扩流/回滚时 → 调整权重 |
| 数据校验 | 客户业务脚本 | 发布完成后 → 触发校验 → 报告结果 |
| 云资源调整 | 云厂商 API | 非代码扩容 → 调用云 API 调整规格 |
| 监控查询 | Prometheus / DataDog API | 验证阶段 → 查询指标 → 判断健康 |
| 构建部署 | 轻雪 DevOps / Jenkins / ArgoCD | 决策后 → 调用对应平台执行 |

### 7.3 插件系统：独立部署的微服务

GweiOps 的插件不是装载在大脑内的代码扩展，而是**独立部署、独立扩展、独立升级的微服务**。

插件通过 gRPC 与大脑通信，可以用任何语言实现，部署在任何位置——同集群、不同集群、不同云、客户内网。大脑通过 gRPC 端点地址调用插件，不在乎插件在哪里、用什么语言实现。

**部署形态：**

| 维度 | 选项 | 说明 |
| :--- | :--- | :--- |
| **位置** | 同集群 / 不同集群 / 不同云 / 客户内网 | 大脑通过 gRPC 端点调用，不在乎插件在哪 |
| **副本数** | 1 个或多个 | 高可用场景可水平扩展，大脑自动负载均衡 |
| **资源规格** | 插件独立配置 | 资源消耗型插件可以单独扩容 |
| **更新策略** | 独立滚动更新 | 插件更新不需要重启大脑 |
| **语言** | 任意支持 gRPC 的语言 | Python、Go、Rust、Node.js、Java 等 |

**插件类型：**

- 资源操作类：K8s 扩缩容、ECS 扩缩容、存储扩容、数据库连接池调整
- 流量调度类：DNS 切换、负载均衡权重调整、降级开关、熔断配置
- 配置变更类：环境变量更新、配置文件热加载、Feature Flag 切换
- 清理恢复类：日志清理、临时文件清理、缓存刷新、服务重启
- 外部联动类：Jira 工单创建、PagerDuty 事件确认、Slack 通知

**插件市场：** GweiOps 官方维护插件市场，社区开发者可提交插件。插件经过安全审核后上架，用户一键安装启用。插件开发者可获得商业版分成。

### 7.4 官方插件包

GweiOps 官方提供预置的插件包，一键安装，开箱即用：

| 包名 | 适用场景 | 包含插件 | 默认策略 |
| :--- | :--- | :--- | :--- |
| **web-service-default** | 通用 Web 服务运维 | 上游追踪、金丝雀发布、K8s 扩容、日志清理 | 优先上游更新，补丁需人工确认，自动金丝雀 |
| **financial-compliance** | 金融行业（高合规要求） | 安全合规检查、审计日志、禁止补丁、双人审批 | 禁止未经审计的补丁，所有变更需双人终审 |
| **ai-inference** | AI 推理服务（GPU） | GPU 扩缩容、模型版本回滚、推理延迟监控 | 优先模型版本回滚而非代码修复，扩缩容阈值更敏感 |
| **startup-fast** | 创业团队（求快） | 上游追踪、自动合并（低风险）、飞书通知 | 低风险 PR 自动合入，仅推送汇总报告 |

插件包之间可以叠加。用户先装基础包，再装合规包，规则自动合并。

### 7.5 优雅停机与滚动升级

GweiOps 的大脑和所有插件都实现同一套优雅停机接口，由大脑统一协调：

- 升级前，大脑向所有组件发送准备停机信号
- 每个组件返回是否有进行中的任务及预估完成时间
- 全部空闲 → 立即升级；有任务进行中 → 等待完成（可配置超时）
- 新版本组件启动后自动注册，大脑将新任务路由到新组件
- 旧组件完成迁移后优雅退出

**升级过程中不丢失任何业务连续性，也不需要人类介入。**

### 7.6 大脑管自己

GweiOps 大脑本身跑在 Kubernetes 里，它的健康状态、升级、回滚、故障恢复，全部由 GweiOps 自己的逻辑处理。

- 大脑自身 Pod 异常重启：状态机从持久化存储恢复，未完成的任务重新调度
- 大脑版本升级：新版本启动后自动检测进行中的任务，继续执行或优雅取消
- 大脑自身响应变慢：观测智能体发现延迟上升，自动水平扩容大脑副本数
- 大脑依赖的外部服务（如 LLM API）不可用：自动降级，使用缓存方案或轻量级修复策略
- 记忆系统存储空间不足：自动触发记忆压缩，推送通知"记忆库已压缩，释放 X% 空间"

**一个不会修自己的运维系统，不值得被信任去修你的生产环境。**

## 八、生态合作

### 8.1 与轻雪 DevOps 的关系

GweiOps 与轻雪 DevOps 是上下游关系，不是竞品关系。

轻雪 DevOps 解决的是"自动化代码→部署流程"的问题——构建、部署、金丝雀发布、回滚的执行引擎。GweiOps 解决的是"如何运维"的决策编排问题——感知问题、诊断根因、决策修复路径、调度执行工具、验证结果、推送审核。

两者结合，用户不需要在工具之间切换。从问题到修复的整个闭环由 GweiOps 的大脑统一驱动，具体动作交给轻雪 DevOps 执行。GweiOps 做闭环的大脑，轻雪 DevOps 做工具的执行。

GweiOps 同样可以对接 Jenkins、ArgoCD、GitHub Actions 等其他执行工具。轻雪 DevOps 是 GweiOps 可以对接的执行层之一，也是最紧密的合作伙伴。

### 8.2 与其他工具的关系

GweiOps 不与任何现有工具竞争。它通过 API 对接所有工具：

- 代码托管：GitHub、GitLab、Gitee
- CI/CD：轻雪 DevOps、Jenkins、ArgoCD、Flux、GitHub Actions
- 监控：Prometheus、Sentry、DataDog
- 服务网格：Istio、Linkerd
- 通知：飞书、钉钉、Slack
- 云厂商：AWS、阿里云、腾讯云

**GweiOps 是缝合者，不是替代者。**


## 九、开源战略

### 9.1 为什么开源

**信任。** 运维系统涉及企业最核心的代码和基础设施。开源让每一行代码可审查、可审计，客户部署的是自己看得懂的软件。

**生态。** 社区贡献适配器和插件——有人写 Gerrit 适配器，有人写 Gitee 适配器，有人写扩容插件，有人写流量调度插件。贡献越多，GweiOps 覆盖越广，对每个客户的价值越大。

**护城河。** 真正的护城河不是代码，而是跑在客户仓库里的决策记忆和围绕 GweiOps 形成的社区。代码可以 fork，记忆和社区 fork 不走。

### 9.2 开源策略

核心代码采用 AGPL 许可证发布，适配器层和插件接口采用 MIT 许可证。核心引擎、多智能体系统、抽象层、状态机、Dashboard、插件管理器全部开源。


## 十、社区共建与自托管实践

### 10.1 GweiOps 跑在 GweiOps 上

GweiOps 团队自身的开发流程和生产环境，是系统最真实、最严苛的测试场。

GweiOps 的 GitHub 仓库向社区开放。所有外部提交的 Issue 和 PR，由 GweiOps 自己的 AI Agent 自动分流、打标签、初步审查。社区贡献者提交代码后，GweiOps 自动触发 CI 构建、测试和金丝雀发布。

GweiOps 官方 SaaS 服务本身也跑在 GweiOps 的守护之下——监控告警由 GweiOps 接收并处理，服务异常时自动诊断、修复、金丝雀发布。配额管理、计费异常、新功能灰度等内部运维事务，全部由 GweiOps 自己的智能体接管。

**GweiOps 团队是产品的第一个也是最严格的用户。每一次社区 Issue 的处理、每一次服务的自愈，都是系统能力的公开演示。**

### 10.2 自托管作为最佳实践

GweiOps 仓库中公开了自用的配置文件——如何处理自己的 Issue、PR 审核策略、金丝雀规则、扩容策略。社区成员可以 Fork 仓库，观察 GweiOps Agent 在自己仓库中的全部操作记录。当社区用户部署自己的 GweiOps 实例时，他们复制的是已经验证过的生产级实践。

### 10.3 社区贡献的激励

- **适配器贡献**：编写平台适配器，合并后获得一次性奖励
- **插件贡献**：开发运维插件（扩容、流量调度、配置变更等），上架插件市场后获得商业版分成
- **规则库贡献**：提交 Issue 处理规则和修复模板
- **持续贡献者**：受邀成为 Committer，参与核心决策

### 10.4 开源与商业的正向循环

SaaS 和私有化部署的营收中，固定比例投入开源研发——支付核心维护者薪酬、社区贡献者奖励、基础设施成本。开源是信任的基石，商业化是可持续的保障。两者之间不存在零和博弈。


## 十一、商业模式与定价

### 11.1 三种交付形态

| 形态 | 说明 | 适用场景 |
| :--- | :--- | :--- |
| **SaaS 云服务** | 一键式托管，注册即用，包含官方插件包 | 中小团队，快速上手 |
| **开源社区版** | 完全免费源代码，包含大脑核心 + 内置能力 | 个人开发者、开源项目 |
| **私有化企业版** | 专属部署，离线运行，插件可私有化或 SaaS | 金融、军工、政务 |

### 11.2 定价模型

SaaS 云服务按团队规模阶梯定价。基础版面向五人以下小团队，包含核心功能和有限 API 调用，包含常用插件包。专业版包含全部功能、无限 API、SLA 保障和全部插件包。

私有化企业版采用一次性授权费加年度维护费，按适配平台数量、集群规模和功能模块报价，支持完全离线部署。

开源社区版完全免费，无任何隐藏费用。

插件市场中的社区插件，部分免费，部分由开发者定价。GweiOps 平台从付费插件的交易中抽取一定比例作为平台维护费和社区基金。

### 11.3 商业与开源的边界

开源与企业版共享同一套核心代码。区别在于交付形态和服务层级，而非功能裁剪。商业收入用于维持核心团队、持续投入开源研发和社区运营。

## 十三、结论

GweiOps 不是一个更好的代码生成工具，也不是一个更智能的 CI/CD 系统。

**GweiOps 是一个全新的品类——AI 驱动的 DevOps 全链路运维大脑。**

它从代码仓库 Issue、安全扫描器、CVE 漏洞源、监控告警系统中感知问题，自主决策采用代码修复、上游更新还是非代码运维动作，完整执行从诊断到验证的闭环流程。夜间无人值守自动处理，人类早晨从容审案。

它与 K8s 分层协作——K8s 调度容器，GweiOps 调度智能体任务。它是运维工具链的缝合者——把 Git、CI/CD、监控、K8s、Service Mesh、云厂商 API 缝合成一个统一的运维决策界面。

它完全开源，任何人都可以审查、部署、修改、贡献。它提供一键式 SaaS 服务和私有化部署。它用自己管理自己——GweiOps 自己的仓库和 SaaS 服务由 GweiOps 守护，这是最硬核的信任证明。

**GweiOps 解决的不是"运维工具"的问题，而是"如何运维"的问题——把几百人团队的运维知识压缩到一个 AI 大脑中，让几个人就能管住复杂的系统。**
